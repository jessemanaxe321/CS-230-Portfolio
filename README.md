# CS-230-Portfolio

This repository is for my CS 230 Operating Platforms coursework at Southern New Hampshire University.

## Course Overview

In this course, I worked with software design concepts related to operating platforms, system architecture, memory management, storage management, distributed systems, networks, and security. The main project focused on The Gaming Room and its game, Draw It or Lose It.

## Project

The main portfolio artifact for this course is a software design document for Draw It or Lose It. This document includes recommendations for expanding the application from an Android-only game into a web-based application that can run across multiple platforms.

## Skills Demonstrated

- Software design documentation
- Operating platform evaluation
- Client-server architecture
- Distributed systems and networking concepts
- Memory and storage management
- Security planning for web-based applications

## Module Eight Journal Reflection

The client for this project was The Gaming Room. Their intention was to expand upon their game Draw It or Lose It, which was previously only available for Android devices, into a web- or cloud-based application that would allow users to play on multiple platforms. They wanted the software to allow for multiple teams and players, unique game names and team names, and to also restrict the game service so that only one instance is running at any given time. The client also wanted advice on what operating platform would be best, as well as what architecture, storage, memory management, security, and deployment method would work best for their purposes.

One thing that I believe I did well in my documentation was relating the client’s needs to my design decisions. Instead of just listing my recommendations, I tried to provide reasoning behind why those choices made sense. I believe that adding context to the decision-making process can be beneficial to the client as well as the designers and developers who will later be using the document. For instance, I recommended a web-based/cloud-based application because The Gaming Room wanted their game to be able to reach more users on different platforms rather than limiting their software to Android.

Working through a document like this was helpful to me because it forced me to slow down and plan out the application before diving into coding. It allowed me to think about what I wanted the software to do in terms of requirements and limitations. I also figured out what architecture would suit the application, as well as what I needed to do in terms of security, scalability, and user requirements. Once I had those pretty well thought out, it became easier to visualize how I wanted the different pieces of the application to work together.

If I were to revise one aspect of my work, it would be the platform comparisons and recommendation. I would spend more time going into detail about what I liked and disliked about each platform. By providing more information on why my recommendation was better than the others, it would be easier for the client and development team to follow along with my thought process.

I determined the user’s needs by identifying what The Gaming Room wanted the software to accomplish. They needed it to operate on multiple platforms, allow multiple players, and restrict players from using the same game name or team name. I believe that considering the user’s needs is a critical step in software design. You could write the most beautiful code that functions exactly how you want it to, but if you’re not solving the user’s problem, your project was not successful.

I approached designing this software by breaking the client’s needs into different sections that would require design decisions. I compared platforms, decided on an architecture, and figured out what I needed to do to satisfy storage, memory, security, and scalability requirements. If I were to design a similar project in the future, I would take a similar approach by first understanding what the client wants to achieve, then comparing how each decision can affect the user and the future of the application.
